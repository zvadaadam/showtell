# Bundle v3: browser-authored motion

Bundle v3 is Showtell's path for crafted videos. A smart agent authors normal HTML/CSS plus one paused GSAP timeline; Showtell supplies the measured clock, live repo/data inputs, deterministic browser, captions, presenter, audio, review, and encoding.

The split is deliberate:

- The agent owns the visual world and choreography.
- Showtell owns truth, time, and production.

## Bundle shape

```text
my-video.showtell/
  spec.json
  hyperframes/*.html
  assets/**
  compiled-plan.json  # generated by Showtell; never author this
```

`spec.json` stays thin and uses the literal bundle version `3`. It declares narration, repo refs, assets, semantic ranges, theme/chrome policy, and visual entry points. It does not contain pasted source, frame math, or ffmpeg commands.

Every designed v3 bundle scene uses a browser HyperFrame with `visual.kind: "web"`. There is no lower-fidelity bundle renderer. Use renderer-provided web components for straightforward code, diff, and chart scenes, then compose them with normal HTML/CSS/GSAP when the story needs custom choreography, layout, or a declared mix of repo/data/image inputs.

`visual.kind: "screencap"` is the separate recorded-media arm. It plays an existing Showtell capture session with optional clipping and playback compression; it is not a second visual design runtime. Declarative simple specs remain the quick authoring surface and compile their designed scenes to trusted internal browser HyperFrames.

```json
{
  "version": 3,
  "meta": {
    "title": "A deterministic motion score",
    "fps": 30,
    "aspectRatios": ["16:9", "9:16"],
    "theme": { "preset": "ink" },
    "repo": { "path": ".." }
  },
  "audio": {
    "tts": { "provider": "say" },
    "captions": { "mode": "burn-in", "source": "narration" }
  },
  "scenes": [
    {
      "id": "proof",
      "narration": {
        "lines": [{ "id": "l1", "text": "The renderer reads the live implementation and seeks every frame exactly." }]
      },
      "refs": {
        "source": {
          "kind": "code",
          "file": "src/motion.ts",
          "lineStart": 1,
          "lineEnd": 20,
          "focus": [8, 14]
        }
      },
      "visual": {
        "kind": "web",
        "src": "hyperframes/proof.html",
        "props": { "title": "Live bytes. Exact frames." },
        "inputs": { "source": "source", "reveal": "line:l1" }
      }
    }
  ]
}
```

Narration uses measured `duration: "auto"` semantics by default. Compilation synthesizes TTS first, measures it, then resolves every range against the real audio. A positive numeric scene duration is available when an existing media window must be fitted exactly; Showtell distributes its narration-line clock across that explicit span.

For recorded media, point at a capture session instead of an HTML entry:

```json
{
  "id": "demo",
  "narration": {
    "lines": [{ "id": "l1", "text": "The interaction stays focused while inactive gaps compress." }]
  },
  "visual": {
    "kind": "screencap",
    "sessionRef": "checkout_demo",
    "clip": { "start": 1.5, "end": 18.25 },
    "playback": { "mode": "smart", "camera": "auto", "actionEffects": "auto" }
  }
}
```

`sessionRef` is a capture session ID under `.showtell/captures`, never a path. `clip` uses source-video seconds and requires `end > start`. Playback modes are `realtime`, `action-only`, and `smart`; the latter two can use capture event sidecars to compress gaps, follow actions, and add interaction feedback.

## HTML contract

Each HTML entry has exactly one static manifest:

```html
<script type="application/showtell+json">
  {
    "schemaVersion": 3,
    "propsSchema": {
      "type": "object",
      "additionalProperties": false,
      "required": ["title"],
      "properties": { "title": { "type": "string", "minLength": 1 } }
    },
    "inputs": {
      "source": { "kind": "repo", "refKind": "code" },
      "reveal": { "kind": "range" }
    }
  }
</script>
```

Input ports are literal and typed:

- `repo`: a scene `code` or `diff` ref; optionally constrain with `refKind`.
- `asset`: a declared `image`, `audio`, or `data` asset; optionally constrain with `assetType`.
- `range`: a line, beat, scene, named range, or explicit `{from,to}` span.

`visual.inputs` maps ports to declarations. HTML never reads the repo or filesystem itself.

## Deterministic runtime

Before authored code runs, Showtell injects:

- pinned Chromium, Playwright, GSAP, and local font files;
- `window.__showtell`;
- theme CSS variables;
- renderer-owned web components such as `<st-code>`.

Run `showtell bundle components` for the machine-readable current contract and `showtell bundle inspect <dir>` for the exact runtime identity/source hash of a visual. The compiled plan records the Chromium revision, browser version, Playwright version, GSAP version, device scale, locale, timezone, color profile, color scheme, reduced-motion setting, and embedded fonts.

The byte-level guarantee is scoped deliberately: identical inputs rendered on the same supported platform with the same pinned runtime produce identical output bytes. Showtell does not claim byte-identical pixels across different operating systems or architectures; the recorded runtime identity makes that boundary explicit and auditable.

Important runtime fields:

```js
const st = window.__showtell;

st.props; // validated visual.props
st.inputs; // resolved declared ports
st.viewport; // width, height, aspectRatio, fps
st.scene; // id, duration, line metadata, progress
st.line; // active line id/index/progress
st.time; // absoluteMs, sceneMs, frame
st.range("reveal"); // startSec/endSec/durationSec/progress/active
st.random("stable-key"); // deterministic 0..1
st.safeArea; // combined caption/presenter insets
st.presenter; // renderer-owned presenter state and exclusion
st.captions.safeArea; // caption-only exclusion
```

Theme variables include semantic colors/fonts and chrome-safe insets:

```css
var(--st-bg); var(--st-fg); var(--st-subtle);
var(--st-accent); var(--st-accent-2);
var(--st-success); var(--st-warning);
var(--st-surface); var(--st-border);
var(--st-font-display); var(--st-font-body); var(--st-font-mono);
var(--st-safe-top); var(--st-safe-right); var(--st-safe-bottom); var(--st-safe-left);
```

The built-in fonts are Inter, League Gothic, JetBrains Mono, and Space Mono. For a distinctive deterministic pairing, prefer League Gothic for display plus Space Mono or JetBrains Mono for technical copy.

## One paused timeline

Author the fully visible end-state layout in CSS first, then animate into it:

```html
<script>
  const st = window.__showtell;
  const reveal = st.range("reveal");
  const timeline = gsap.timeline({ paused: true });

  timeline.fromTo(
    ".hero",
    { opacity: 0, y: 72, scale: 0.94 },
    { opacity: 1, y: 0, scale: 1, duration: 0.72, ease: "expo.out" },
    reveal.startSec,
  );
  timeline.fromTo(
    ".signal-fill",
    { scaleX: 0 },
    { scaleX: 1, duration: reveal.durationSec, ease: "none" },
    reveal.startSec,
  );

  st.timeline = timeline;
</script>
```

Showtell calls `timeline.pause().seek(sceneMs / 1000, false)` for every encoded frame. There is no playback loop to drift.

Forbidden in authored visuals:

- network requests or external URLs;
- timers and asynchronous timeline construction;
- `Date.now()`, wall-clock state, `Math.random()`, Web Crypto randomness, or input devices;
- CSS animations/transitions, the Web Animations API, or SVG SMIL animation;
- dynamic imports, `eval`, or runtime source loading.

Use `st.time`, semantic ranges, and `st.random(key)` instead. The compiler rejects violations before browser launch; guarded author-script execution, animation cancellation, CSP, and request interception enforce the same policy at runtime.

## Live repo evidence

Use renderer-owned components for source truth:

```html
<st-code input="source" reveal-range="reveal" max-lines="18"></st-code>
```

The component receives only the renderer-resolved bytes and renders file/line context consistently. Changing the referenced repo file and recompiling changes the pixels. Do not paste source into props or HTML.

## The quality workflow

Flexibility alone does not produce a good video. Before HTML, make four decisions:

1. **Message:** one sentence stating what the viewer should understand or feel.
2. **Visual world:** one concrete metaphor (signal path, folding map, assembly line, orbit), not a restyled heading.
3. **Identity:** one theme preset plus an intentional display/technical font pairing.
4. **Depth plan:** background atmosphere, one midground focal visual, and foreground metadata/structure.

For each narration line, name one focal visual and one transformation verb: trace, assemble, compare, route, fold, lock, reveal, settle. A frame should usually contain 8–10 total elements across depth layers, but only one primary idea. Registration marks, rules, signal rails, coordinate labels, and restrained atmospheric glows create production density without adding competing panels.

Avoid the slide-deck tells:

- an eyebrow + title at the start of every line;
- centered stacks floating on a flat background;
- identical card grids;
- every element entering with the same `y + opacity` tween;
- several unrelated evidence panels visible at once.

Titles are chapter openers. After the opener, let the code, path, chart, screenshot, or diagram fill the frame. Time is hierarchy: the first appearance is the primary statement.

For a multi-state HTML visual, transition the states together: outgoing and incoming motion overlap at the same timestamp. Use one primary transition vocabulary repeatedly and one accent for the climax. The final state may settle or fade; intermediate states should not disappear into a dead gap before the next arrives.

## Responsive layout

Author one semantic composition that adapts to both orientations:

- Use `%`, `vw`, `vh`, `clamp()`, grid/flex, and `@media (max-aspect-ratio: 4 / 5)`.
- Use absolute positioning for decoration, not for the main content container.
- Respect `--st-safe-*` so captions and presenter chrome do not cover the focal visual.
- On 9:16, recompose zones vertically; do not scale a 16:9 canvas down.
- Keep display text at video scale (roughly 64–120px landscape, larger in portrait) and technical labels at 18px or above.

Review both aspects. A valid manifest does not prove a responsive layout.

## Agent-first loop

All commands are non-interactive and return JSON with actionable `hint` fields.

```bash
showtell bundle themes
showtell bundle components
showtell bundle templates
showtell bundle runtime
showtell bundle validate my-video.showtell
showtell bundle inspect my-video.showtell
showtell bundle workshop my-video.showtell --out .showtell/workshop --aspect 16:9,9:16
showtell bundle review my-video.showtell --out .showtell/review --aspect 16:9,9:16 --samples 5
showtell bundle render my-video.showtell --out .showtell/out --aspect 16:9,9:16
```

Workshop, review, and render compile automatically. Run `bundle compile` directly only when you need to inspect `compiled-plan.json` without rendering. Use workshop for held-state hierarchy and overflow. Use review for motion: it samples the exact timestamps final render will encode and reports pixel deltas per line/aspect. Inspect the filmstrips; advisory static-motion warnings do not fail a valid bundle.

Before handoff, verify:

- every narration claim is visible;
- every expected moving line changes across review samples;
- code/diff evidence comes from declared refs;
- captions/presenter avoid the focal visual;
- 16:9 and 9:16 are composed, not merely scaled;
- the MP4 duration matches the compiled plan.
